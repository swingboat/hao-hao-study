# 学生一轮复习 MVP 任务清单

> 面向用户：高二学生，目标为 2027 年高考，当前处于数学第一轮复习。
>
> 核心目标：让学生每天打开后不需要自己判断，直接知道“今天练什么、为什么练、做完哪里变强、明天还会补什么”。

## 当前判断

当前 MVP 已经具备admin端内容导入、web端今日学习、练习提交、结果页、进度页和 planner 设置等基础能力。下一阶段不优先扩展 AI 对话、复杂推荐器或多学科，而是把“一轮复习每日闭环”打透。

## 执行原则

- web端文案只使用学习语言，不展示内部 ID、provider、pool、fallback、Job 等工程状态。
- 每个任务都必须能独立演示，优先纵向切片：数据准备、业务逻辑、UI、测试一起完成。
- LLM 解析质量、prompt/schema 调整不在本仓库直接试验；如需要变更，先到 `how-to-use-llm-proxy` 验证后同步。
- 按 worktree 边界执行：`packages/**`、`docs/**` 走 main；`apps/admin/**` 走 `worktrees/admin`；`apps/web/**` 走 `worktrees/web`。

## 里程碑 A：真实学生闭环验收

### A1. 建立三天学习验收脚本

**类型**：AFK  
**优先级**：P0  
**建议 worktree**：main + web  
**依赖**：当前基线  
**实施文档**：`docs/Student_First_Round_MVP_A1_Three_Day_Acceptance.md`

**要完成什么**

准备一个固定验收场景：学生 `niki` 以高二一轮复习身份连续完成 3 天学习。每一天都能从首页开始，完成一组练习，提交后看到解析和进度变化。

**验收标准**

- [ ] 有一份中文验收清单，覆盖“登录 → 今日学习 → 作答 → 提交 → 解析 → 首页进度变化”。
- [ ] 第 1 天完成后，首页能看到至少一个知识点从“未开始”进入学习状态。
- [ ] 第 2 天能召回第 1 天的错题或间隔复习内容。
- [ ] 第 3 天能看到本周完成次数和章节进度变化。
- [ ] web端页面不出现内部 ID、provider、pool、fallback、Job 等工程词。

### A2. 补齐web端端到端测试

**类型**：AFK  
**优先级**：P0  
**建议 worktree**：web  
**依赖**：A1  

**要完成什么**

为学生主旅程补端到端测试或等价的集成测试，确保每日学习闭环不会被后续改动打断。

**验收标准**

- [ ] 未登录访问学生页会跳登录。
- [ ] 已登录学生能开始今日学习并进入答题页。
- [ ] 提交答案后写入 attempt，关闭 session，并进入结果页。
- [ ] 结果页展示学生可理解的答案、解析、对错和知识点变化。
- [ ] 测试覆盖空题库或无可练内容的正向学习文案。

## 里程碑 B：错题修复闭环

### B1. 学生错题本列表

**类型**：AFK  
**优先级**：P0  
**建议 worktree**：web  
**依赖**：A1  

**要完成什么**

增加学生可访问的错题本入口，按知识点展示仍需攻克的错题。学生看到的是“需要再练的知识点和题目”，不是数据库状态。

**验收标准**

- [ ] 首页有“错题复习”入口。
- [ ] 错题本只展示当前学生自己的 `open` 错题。
- [ ] 错题按知识点分组，显示题干摘要、最近错误时间、累计错误次数。
- [ ] 不展示 `mistake_book_entry`、`question_id`、`kp_id` 等内部字段。
- [ ] 无错题时显示正向文案，例如“当前没有需要回炉的错题，继续保持今天的节奏”。

### B2. 单题错题重做

**类型**：AFK  
**优先级**：P0  
**建议 worktree**：web + packages/shared  
**依赖**：B1  

**要完成什么**

学生从错题本选择一道题，进入单题重做流程。提交后立即看到对错、解析，并更新错题状态。

**验收标准**

- [ ] 学生能从错题本进入单题重做页。
- [ ] 单题提交后立即显示“你的答案 / 正确答案 / 解析”。
- [ ] 连续做对达到规则阈值后，该错题从错题本移除。
- [ ] 答错后保留在错题本，并更新错误次数或最近练习时间。
- [ ] 所有写库操作保持事务边界，避免只写 attempt 但未更新错题状态。

### B3. 今日学习结果页显示错题消解

**类型**：AFK  
**优先级**：P1  
**建议 worktree**：web  
**依赖**：B2  

**要完成什么**

当学生在今日学习中做对历史错题并达到消解规则时，结果页明确告诉学生“哪道错题已经攻克”。

**验收标准**

- [ ] 结果页能展示本次攻克的错题数量。
- [ ] 对应题目卡片显示“已从错题本移除”一类学习语言。
- [ ] 没有攻克错题时不展示空洞提示。

## 里程碑 C：一轮复习首页

### C1. 首页改成“今日复习任务台”

**类型**：HITL  
**优先级**：P0  
**建议 worktree**：web  
**依赖**：A1  

**要完成什么**

首页首屏直接回答学生的四个问题：今天练什么、为什么练、预计多久、做完能看到什么变化。

**验收标准**

- [ ] 首页主标题围绕“今日一轮复习”或等价学习语言。
- [ ] 展示今日推荐的章节/知识点摘要。
- [ ] 展示推荐原因，例如“巩固新学内容”“回炉最近错题”“安排到期复习”。
- [ ] 展示预计题量和预计用时。
- [ ] 主按钮明确，例如“开始今日复习”。

**需要人工确认**

需要先确认首页最终文案风格：更偏“克制学习工具”还是“略带陪伴感的学习助手”。

### C2. 移除web端 planner 工程化设置

**类型**：HITL  
**优先级**：P1  
**建议 worktree**：web + packages/shared  
**依赖**：C1  

**要完成什么**

审视当前 planner 设置页。高二学生不应该直接调权重、pool 或推荐器配比。若保留设置，只保留学生能理解的模式。

**验收标准**

- [ ] web端不展示“权重”“pool”“new_knowledge”“spaced_review”等工程词。
- [ ] 如保留设置，选项仅为学习语言，例如“正常复习”“时间紧，少做一点”“优先攻克错题”。
- [ ] 设置不会破坏一轮复习主线，不允许学生完全关闭错题或间隔复习。

**需要人工确认**

需要确认是否暂时隐藏 planner 设置，还是转译成 2-3 个学习模式。

## 里程碑 D：一轮复习进度地图

### D1. 章节进度地图

**类型**：AFK  
**优先级**：P0  
**建议 worktree**：web + main  
**依赖**：A1  

**要完成什么**

把当前进度页打磨成一轮复习地图。学生应该能按章节看到覆盖情况：未开始、需要加强、学习中、已掌握。

**验收标准**

- [ ] 进度页按教材章节分组。
- [ ] 每章显示知识点数量、已掌握数量、需要加强数量。
- [ ] 每个知识点显示学生语言状态，不显示 mastery 原始分值作为主信息。
- [ ] 支持只看“需要加强”的知识点。
- [ ] 仅展示学生已解锁范围内的知识点。

### D2. 今日学习后进度变化提示

**类型**：AFK  
**优先级**：P1  
**建议 worktree**：web  
**依赖**：D1  

**要完成什么**

学生做完一组题后，结果页要明确反馈“本次哪些知识点有变化”，建立持续使用的正反馈。

**验收标准**

- [ ] 结果页展示本次提升的知识点。
- [ ] 结果页展示本次仍需加强的知识点。
- [ ] 返回首页或进度页后，变化能被看见。

## 里程碑 E：冷启动诊断

### E1. 首周诊断模式设计

**类型**：HITL  
**优先级**：P1  
**建议 worktree**：docs + main  
**依赖**：A1  

**要完成什么**

设计轻量冷启动诊断：不是正式考试，而是用少量代表题快速建立初始掌握度，帮助高二学生知道一轮复习从哪里开始补。

**验收标准**

- [ ] 明确诊断只覆盖已导入题库和已解锁知识点。
- [ ] 明确每章抽多少题、总题量上限、预计用时。
- [ ] 明确诊断结果如何初始化 mastery。
- [ ] 明确诊断结束后如何回到每日学习主线。

### E2. 实现冷启动诊断入口

**类型**：AFK  
**优先级**：P1  
**建议 worktree**：web + packages/shared  
**依赖**：E1  

**要完成什么**

当学生历史学习数据不足时，首页优先引导完成一次轻量诊断。诊断完成后，系统再进入正常每日复习节奏。

**验收标准**

- [ ] 新学生首页看到“先完成一次基础诊断”的入口。
- [ ] 诊断题量有限，不超过一个正常 session 的心理负担。
- [ ] 诊断提交后能更新初始掌握状态。
- [ ] 已完成诊断的学生不再被反复打断。

## 里程碑 F：题库水位与运营保障

### F1. 知识点题量水位检查

**类型**：AFK  
**优先级**：P0  
**建议 worktree**：main + admin  
**依赖**：当前基线  

**要完成什么**

admin端需要知道哪些知识点题量不够。对一轮复习来说，题库不足比推荐算法不够聪明更致命。

**验收标准**

- [ ] 每个已发布知识点能统计可用题数量。
- [ ] 低于水位的知识点有清晰标记，例如“题量不足，建议补到 5 题”。
- [ ] web端遇到题量不足时，文案仍然正向，不暴露“题库不够”。

### F2. 学生可练范围健康检查

**类型**：AFK  
**优先级**：P0  
**建议 worktree**：main + admin  
**依赖**：F1  

**要完成什么**

检查某个学生当前解锁范围内是否有足够题目支撑连续 3 天学习。

**验收标准**

- [ ] admin端学生列表或详情能看到“可练知识点数 / 可用题数 / 低水位知识点数”。
- [ ] seed 学生 `niki` 能通过健康检查。
- [ ] 健康检查结果使用运营语言，不展示底层 SQL 或内部数组。

## 里程碑 G：解析质量与学习语言

### G1. 解析展示质量抽检

**类型**：HITL  
**优先级**：P1  
**建议 worktree**：docs + web  
**依赖**：A1  

**要完成什么**

用真实学生视角抽检解析页：学生能不能看懂“为什么错、下次怎么办”。本任务不直接改 LLM prompt，只记录展示层和内容质量问题。

**验收标准**

- [ ] 抽检至少 20 道题的解析展示。
- [ ] 记录每道题是否包含：考点、关键步骤、易错点、最终答案。
- [ ] 展示层能正确渲染公式、选项、题图。
- [ ] 需要 LLM 能力改动的问题单独整理给 `how-to-use-llm-proxy` 验证。

### G2. web端文案去工程化

**类型**：AFK  
**优先级**：P0  
**建议 worktree**：web  
**依赖**：A1  

**要完成什么**

全量扫描web端页面，把工程状态转译为学生/家长能理解的学习语言。

**验收标准**

- [ ] web端不直接展示数据库主键、provider id、job id、enum 原值、内部路径。
- [ ] 空状态、错误态、题量不足态都使用正向学习语言。
- [ ] 题干、答案、解析统一走 `@hao/llm` 的展示格式化方法。

## 建议实施顺序

1. A1：建立三天学习验收脚本。
2. G2：web端文案去工程化。
3. B1 + B2：错题本列表和单题重做。
4. C1：首页改成今日复习任务台。
5. D1：章节进度地图。
6. F1 + F2：题库水位与学生可练范围健康检查。
7. A2：补齐端到端测试。
8. B3 + D2：结果页强化反馈。
9. E1 + E2：冷启动诊断。
10. G1：解析质量抽检并形成 LLM 能力改进输入。

## 第一轮建议只启动的任务

为了尽快得到学生可用反馈，第一轮建议只做这 5 个：

- [ ] A1：建立三天学习验收脚本
- [ ] G2：web端文案去工程化
- [ ] B1：学生错题本列表
- [ ] B2：单题错题重做
- [ ] C1：首页改成“今日复习任务台”

这 5 个完成后，就可以让高二学生真实连续使用 3 天，判断产品是否真正解决一轮复习的日常问题。
