# Q1-Q10 全景设计审视与冲突分析

> 智能学习助手产品架构完整决议总结 + 跨Q逻辑冲突识别

---

## 第一部分：Q1-Q10 详细决议总结

### Q1 目标体系与考纲模板架构

**核心问题**：学生目标如何分层管理，考纲模板如何成为系统的知识底座。

**关键决议清单**：

1. **两层目标分离**
   - Goal Template（运营资产）：由教育部课标 + 考纲权重组成，复用性强
   - Goal Instance（学生数据）：模板 + 学生起点 + 截止时间 + 日预算的组合
   - 必须在数据模型阶段分离，为版本迁移预留字段

2. **学生最多维护一个一层目标**，但可跨学科
   - 同科目多个专项（二层目标）同时最多 1 个 active
   - 专项遵循西蒙学习法，走平行赛道不影响主线

3. **二层目标对主 session 的影响**
   - 平行赛道设计：学生在"今日学习"和"专项训练"间主动切换
   - 专项完成必须强制结算（设硬性截止），不能无限期挂着

4. **决策优先级框架（三层）**
   - 第一层必做配额：艾宾浩斯到期项当天必清
   - 第二层主线驱动：按阶段分流（新学期→学校进度；复习期→考纲权重）
   - 第三层弱点补强：剩余时间填充

5. **阶段判定维度**
   - 学科独立：数学和英语的阶段进度不耦合
   - 对内连续值（0-1表示"紧迫度"），对外枚举展示（新学/巩固/一轮/二轮/冲刺）

6. **课标锚点获取与模板数据源**
   - 课标锚点：每条"内容要求"映射唯一标准文件位置
   - 4类数据源缺一不可：课标、高考评价体系、卷子结构、历年真题
   - MVP采购清单1000元以内，人工工作量30-45人天

7. **教研角色与版本迁移**
   - MVP考纲从"成熟教辅+AI抽课标+自审"混合而来
   - 教研专家缺席是已知高风险，列为融资前置依赖
   - 模板版本号字段MVP必须预留（不做迁移功能但预留字段）

**重要的边界条件**：

- MVP数学模板精确到：**新课标I卷数学 + 安徽等适用省份**
- 知识点粒度：初高中单学科200-400个（太细则AI标注无效，太粗则诊断无意义）
- 知识点DAG而非树（一个知识点可多个前置）
- 考点定义 = 知识点 × 能力点 × 题型场景的笛卡尔积

### Q3 学生掌握度判定与讲述题

**核心问题**：什么时候算"学会了"，如何用费曼讲述验证真掌握。

**关键决议清单**：

1. **掌握度判定的最小单位：知识点**
   - 对学生展示：挂在知识点层级（避免考点级的认知负担）
   - 系统内部：同时维护知识点和考点两层（推荐引擎用考点级精准采样）

2. **掌握度数据类型**
   - 内部存连续值（0-1）
   - 对外展示4档枚举：未开始/学习中/熟悉/已掌握
   - 衰减规则预留但v1版暂不启用

3. **讲述题（费曼的产品化）**
   - 产品形式：不是独立模式，而是主线session中的一种特殊题型
   - 触发条件（三同时满足）：
     - 该KP mastery在0.6-0.85
     - 距上次该KP讲述≥7天
     - 该KP未feynman_verified
   - 频率：每session最多1道，避免打扰

4. **讲述题评判规则（三档）**
   - 未达标（<30字或无关）：不算过，mastery不变，建议重讲
   - 达标（覆盖一半关键点）：算过，mastery+0.15
   - 优秀（准确完整）：算过，mastery+0.20

5. **答题流程重设计（Q3.4决议）**
   - 固定时长25分钟+题量自适应填满
   - 一次性提交（无中间LLM延迟）
   - 提交后批量AI分析一次
   - 学生可选"训练相关知识点"触发5道二级题目

### Q5 错题本、根因分类、拍照分析

**关键决议清单**：

1. **错题根因的6类分类**
   - 概念性错误→推前置题+KP学习材料
   - 方法性错误→推同KP变种题
   - 审题错误→推审题专项
   - **计算粗心→不进错题本，仅计数**
   - 时间压力→推限时重做
   - 超纲/超难→不再做，标archived

2. **错题本字段与状态**
   - ERROR_LOG表：item_id, student_id, root_cause, error_count, status(open/resolved)
   - computational错误不进错题本主体（污染推荐）
   - 下架题标invalidated（不计入错题统计）

---

### Q6 题目正确性边界、校验、灰度

**关键决议清单**：

1. **题目来源策略**
   - B为主：题库召回（真题/联考/质检，MVP500-1000道起步）
   - C补充：题库召回+LLM参数改写（仅改数字/参数/情境）
   - A不做：纯LLM生成

2. **灰度池机制**
   - LLM改写题→灰度池（5-10%学生，7天观察）
   - 期间举报率<1% 且 ≥20人做过→进主池
   - 期间≥2次举报→自动下架进复审队列

3. **学生举报机制与下架**
   - 同一学生对同一题仅计1次举报
   - 主池3次举报下架，灰度池2次举报下架
   - 下架题对所有学生mastery和错题本回滚

---

### Q7 推荐引擎与7池架构

**关键决议清单**：

1. **7个池的设计与优先级**
   - 池1 艾宾浩斯到期池：优先级最高
   - 池2 错题本变种池：高优先级
   - 池3 新知识点池：中优先级
   - 池4 讲述题池：中优先级（按触发条件）
   - 池5 综合题池：v1.5才做
   - 池6 挑战题池：可选
   - 池7 灰度试用池：横向叠加

2. **艾宾浩斯调度规则**
   - 间隔序列：1→3→7→15→30天
   - 临考压缩：距高考≤30天时间隔×0.5
   - 知识点级调度去重

3. **新知识点池的难度自适应**
   - mastery<0.2→难度1-2基础题
   - mastery 0.2-0.5→难度2-3标准题
   - mastery 0.5-0.85→难度3-4进阶题

4. **配比表达式引擎**
   - default配比：new_knowledge 30% / error_book 30% / ebbinghaus min(40%)
   - pre_exam_30days：ebbinghaus 60% / error_book 30% / new 10%
   - pre_exam_7days：ebbinghaus 100%（仅mastery≥0.85）

### Q8 冷启动与章节进度

**关键决议清单**：

1. **冷启动不做摸底卷**
   - 所有KP mastery从0开始演化
   - 入驻采集5必填项：年级、目标考试、章节进度、日时长、省份+教材

2. **第1周配比**
   - 100%新知识点池，难度自适应敏感度调高

3. **第2周配比**
   - new 50% / error_book 30% / ebbinghaus 20%

4. **章节进度管理**
   - 更新方式：每周日主动提示
   - 未学范围KP硬边界：不进任何池

---

### Q9 用户画像与个性化

**关键决议清单**：

1. **MVP画像范围**
   - 做：基础属性、学习能力、抗拒题型
   - 不做：心理动机、偏好难度/时段

2. **抗拒题型应用**
   - 降权但不剔除（出现频率×0.5）

3. **激励机制**
   - 做：streak + 周报 + 庆祝 + 每日量提示
   - 不做：排名 / 徽章 / 家长报告

### Q10 运营侧实体与工作流

**关键决议清单**：

1. **题目状态机（6态）**
   - draft → pending_review → gray/published/deprecated
   - published → under_review → published/draft/deprecated

2. **ITEM_AUDIT_LOG必须有**
   - 记录每次状态变更（who/when/why/from/to）

3. **KP / EXAM_POINT分两层**
   - KP是"概念"（200-300个）
   - EXAM_POINT是"概念×能力×场景"（500-800个）

4. **举报与工作流**
   - 同一学生对同一题仅计1次举报
   - 达阈值先下架再审

## 第二部分：跨Q逻辑冲突 / 矛盾 / 设计错误

### 高严重性冲突（必须修正才能上线）

#### 冲突1：Q3讲述题的mastery跳变规则 vs Q7艾宾浩斯池的触发边界

**冲突点**：
- Q3：讲述题达标时mastery+0.15或+0.20
- Q7：艾宾浩斯到期池触发基于"上次review≥M天"

**问题**：同一KP同一session是否可能既出讲述题又出艾宾浩斯题？

**修正建议**：
```yaml
讲述题池与艾宾浩斯到期池的互斥规则：
- 同一KP在session内不能既出讲述题又出艾宾浩斯题
- 若讲述题触发，该KP的艾宾浩斯到期项延后
- 讲述题通过后mastery跳变，下一session重新评估艾宾浩斯触发
```

---

#### 冲突2：Q6下架题的mastery回滚 vs Q3 mastery演化的连续性

**冲突点**：
- Q6：下架题需回滚所有学生的mastery和错题本
- Q3：mastery演化有历史依赖（mastery(t+1) = f(mastery(t))）

**问题**：撤销过去的答题贡献会导致mastery突然下降，学生看到"掌握度减少"会困惑。

**修正建议**：
```
Q3必须补充完整的mastery演化规则：
- 正确题：mastery(t+1) = mastery(t) + (1 - mastery(t)) * δ_correct (默认0.1)
- 错题（非粗心）：
  - mastery < 0.3：mastery不变
  - 0.3 ≤ mastery < 0.6：mastery → mastery * 0.95
  - mastery ≥ 0.6：mastery → mastery * 0.9
- 讲述题达标：mastery(t+1) = mastery(t) + (1 - mastery(t)) * δ_feynman (0.15 or 0.20)
- 下架题回滚：严格反向演化上述公式
```

---

#### 冲突3：Q5错题根因推断精度 vs Q7变种题召回链路 vs Q3讲述题触发条件的闭环

**冲突点**：
- Q5根因推断是AI判断，精度≤85%
- Q7根据不同根因调用不同召回链路
- Q3讲述题可能作为错题本的攻克路径
- 当AI根因判错时，后续推荐链路完全错位

**问题**：错题根因推断精度不足导致推荐路径偏离，需要闭环反馈机制。

**修正建议**：
```
增加根因推断的反馈闭环：

1. 错题根因初判后，让学生确认或选择候选根因（2-3个）
2. Q7根据学生确认的根因调用对应链路
3. 后续3-5道题的表现反馈到Q5的根因评估模型
4. Q3讲述题的触发规则补充：
   - 若KP近期根因标记中>50%是conceptual，讲述阈值从0.6提到0.7+
```

**风险等级**：**最高严重** - 涉及推荐质量和学习路径的核心

---

### 中严重性冲突（建议在MVP修正）

#### 冲突4：Q8新知识点池的边界 vs 学生超前学习 vs 错题本的"范围问题"

**冲突点**：
- Q8：未学范围KP不进任何池（硬边界）
- Q8补充：学生可主动解锁未学章节
- Q5实现：错题本来自"已学范围"，但学生可能在辅导班提前学过

**问题**："已学范围"的定义不清，导致错题本边界模糊。

**修正建议**：
```
定义"已学范围"优先级：
学校当前进度 > 学生自选超前解锁范围 > 学生历史做过最高KP

上传功能中超界题的处理：
- 若题目KP超出已学范围：不计入error_book_variant
- 学生在上传确认时弹窗提示
- 若学生坚持做超界题，打source=homework_overflow标记
```

---

#### 冲突5：Q3临考期讲述题触发 vs Q9阶段化激励 vs Q7配比临考压缩

**冲突点**：
- Q3：讲述题触发条件包括"mastery在0.6-0.85"
- Q7：临考期≤30天配比变为ebbinghaus 60%（大幅提升复习比重）
- Q9：临考期切换为"焦虑感"激励

**问题**：讲述题可能挤压掉关键的艾宾浩斯复习时间，用户体验不符合预期。

**修正建议**：
```yaml
阶段化讲述题触发规则：
normal_period (>30天)：
  trigger_when: mastery in [0.6, 0.85] AND days_since_last >= 7
  max_per_session: 1

pre_exam_30_days (30-7天)：
  trigger_when: mastery in [0.65, 0.85] AND days_since_last >= 14
  max_per_session: "if(high_mastery_kp > 10, 1, 0)"

pre_exam_7_days (<7天)：
  trigger_when: false  # 禁用讲述题，冲刺期只复习已掌握
```

### 低严重性冲突（v1.5注意即可）

#### 冲突6：Q3每session最多1讲述题 vs 时间预算

**冲突点**：讲述题（打字50-150字）预期需要2-4分钟，但25分钟session要塞10-15道题。

## 第三部分：设计盲区

### 盲区1：mastery初值问题与Q8冷启动的交点

当前mastery=0时的演化假设学生是"真正初学"，但实际上学生可能已听过课。建议在冷启动中补充选项让学生标记"已学过"可初值+0.2-0.3。

### 盲区2：错题根因推断的"跨域问题"

同一道题的多步骤可能有不同根因（如第1步conceptual、第2步computational、第3步methodological），但当前仅支持单一root_cause字段。建议增加sub_errors数组记录详情。

### 盲区3：Q7推荐引擎的"召回不足"应急

题库不足时的递级补充规则未细化，建议明确：单池不足→用其他KP补→全池不足→灰度池补→极端情况出现提示。

### 盲区4：Q6下架题的"数据恢复"

下架题没有恢复逻辑，建议ITEM_AUDIT_LOG中增加is_reversible字段和reversal_deadline（如30天内可手动恢复）。

### 盲区5：Q9缺少"学习持久性指标"

当前画像缺少persistence_score、time_pattern、fatigue_indicator等长期学习指标，建议后台埋点收集但v1.5再决定是否用于调度。

## 最严重的三个冲突总结

1. **冲突1（Q3+Q7）**：讲述题mastery跳变与艾宾浩斯调度的调度冲突
   → 需要明确同KP同session的互斥规则

2. **冲突3（Q5+Q7+Q3）**：错题根因推断精度与推荐闭环的精度问题  
   → 需要增加学生确认和反馈机制

3. **冲突2（Q6+Q3）**：下架题回滚与mastery演化规则的数据一致性
   → 需要完整的mastery演化公式定义和反向回滚逻辑

这三个冲突必须在MVP启动前完全解决，否则上线后会导致推荐失效和学生体验严重偏差。
