**《课程标准》是什么**

全称叫**《普通高中课程方案和课程标准》**和**《义务教育课程方案和课程标准》**，是教育部颁布的国家级文件，规定了**每个学科在每个学段，学生必须达到的知识掌握和能力发展要求**。它是法定的、强制性的教学依据。

关键特征：
- **由教育部颁布**，不是考试院出的。
- **覆盖整个学段**（义务教育 1-9 年级、普通高中 10-12 年级），不只针对升学考试。
- **按学科分册**：每个学科一本，比如《义务教育数学课程标准（2022 年版）》《普通高中物理课程标准（2017 年版 2020 年修订）》。
- **管的是"教什么、学到什么程度"**，不直接管"考什么"。

当前在用的两套：
- 义务教育：**2022 年版**（2022 年 4 月颁布，2022 秋季学期起执行）
- 普通高中：**2017 年版，2020 年修订**（高考 3+1+2 / 3+3 改革的配套）

每本课标的内部结构大致是：
1. 课程性质与基本理念
2. **核心素养**（这是新课标的灵魂——比如数学的"数学抽象、逻辑推理、数学建模、直观想象、数学运算、数据分析"六大核心素养）
3. 课程目标
4. **课程内容**（按主题/模块组织，写明每个主题的"内容要求""学业要求""教学提示"）
5. **学业质量标准**（按水平 1-5 描述学生应达到的表现，高考命题主要锚定水平 4-5）
6. 实施建议（教学、评价、教材编写）

**先讲 DAG 是什么**

DAG = **Directed Acyclic Graph**，中文叫**有向无环图**。三个字拆开看：

- **Graph 图**：由"节点（Node）"和"边（Edge）"组成的结构。
- **Directed 有向**：边是有方向的箭头，A → B 表示"从 A 到 B"，不能反过来走。
- **Acyclic 无环**：顺着箭头走，永远回不到起点。不存在 A → B → C → A 这种循环。

举个生活例子：你做早饭的步骤。
- 烧水 → 泡咖啡
- 切面包 → 烤面包 → 涂果酱
- 烤面包 和 泡咖啡 之间没有依赖

把这些画出来，就是一个 DAG：有方向（必须先烧水才能泡咖啡），无环（你不会涂完果酱再回去烧水）。这种结构能告诉你：哪些步骤必须按顺序，哪些可以并行。

---

**对比一下：树、图、DAG**

| 结构 | 特征 | 例子 |
|---|---|---|
| 树（Tree） | 每个节点只有一个父节点，从根到叶单一路径 | 公司组织架构、文件夹目录 |
| DAG | 节点可以有多个父节点，但不能成环 | 课程先修关系、任务依赖、家谱（祖先方向） |
| 一般图 | 边无方向或可成环 | 社交网络的"好友关系"、地铁线路图 |

**关键差异**：树里一个孩子只能有一个父亲；DAG 里一个孩子可以有多个父亲。这就是为什么知识点要用 DAG 不能用树。

---

**为什么知识点要用 DAG 不能用树**

举数学的例子。"一元二次方程求根公式"这个知识点，它依赖什么？

- 依赖"一元二次方程的概念"
- 依赖"平方根"（公式里有根号）
- 依赖"配方法"（求根公式的推导基础）
- 依赖"代数式运算"

你看，**它有 4 个前置知识点**。如果你硬要用树，就只能选一个当父节点，剩下三个依赖就丢了——系统不知道学生卡在"求根公式"时，可能是因为"平方根"没掌握。

再举一个跨章节的例子。"二次函数与一元二次方程的关系"这个考点：
- 依赖"二次函数图像"
- 依赖"一元二次方程求根"
- 依赖"数形结合思想"

这三个前置在教材里分布在不同章节，甚至不同年级。树结构根本表达不了。

但它一定无环：你不会出现"A 是 B 的前置，B 又是 A 的前置"这种循环——如果出现，说明你的知识点切分错了，需要合并或拆开。

---

**DAG 在你的产品里能做什么**

这不是炫技，是有具体业务价值的：

1. **诊断溯源**：学生在"求根公式"上连续错，系统沿着 DAG 反向查它的前置，发现"配方法"也错过，就能定位真正的根因不是公式本身，是配方法没掌握。这就是你最初问的"学生是知识点没掌握还是能力不行"的答案之一。

2. **学习路径生成**：学生要学"二次函数综合题"，系统沿 DAG 反向遍历，自动列出所有未掌握的前置知识点，按拓扑顺序排出学习计划。这就是"AI 推荐策略"在 Q1 里最具体的落地之一。

3. **阻塞检测**：如果学生某个知识点 A 没过，所有以 A 为前置的下游知识点都应该被标"暂时无法推进"，避免推荐引擎给学生推他根本学不动的题。

4. **跨学科 / 跨章节复用**：同一个知识点（比如"函数图像"）在不同章节、不同学科（数学函数、物理运动学图像）都被引用，DAG 天然支持"多父节点"，不会出现重复录入。

5. **教材版本对齐**：人教版和苏教版章节顺序不同，但知识点 DAG 是共享的。教材只是 DAG 的一种**线性化呈现顺序**。这就是上一轮你和 GPT 讨论的"教材是内容索引层"的真正含义——教材是 DAG 的一条遍历路径而已。

**实际录入时的两个坑，提前告诉你**

1. **粒度陷阱**：知识点切太细，DAG 节点上千，前置关系成指数爆炸，运营录不动；切太粗，诊断溯源失效（学生卡在"函数"上你不知道具体卡哪里）。我的经验值：**初高中单学科 200-400 个知识点是合理量级**，再细就要拆成"知识点 + 子能力"两层。

2. **前置关系本身有强弱**：A 是 B 的"必备前置"（没 A 就完全做不了 B）vs"推荐前置"（没 A 也能学 B，但会更吃力）。如果不区分，诊断溯源会乱报警。建议边上加一个强度字段（强依赖 / 弱依赖）。

**"考点"有没有官方定义？**

所以"考点"的本质是**教研行业为了把课标条目落地到可命题、可讲解、可练习的颗粒度，自己加工出来的中间层**。每家教辅、每个教研团队切出来的考点表都不完全一样。

**对你产品的含义**：

**那为什么我把考点定义成"知识点 × 能力要求 × 题型场景"？**

因为单独的"知识点"颗粒度太粗，没法直接对应一道题。举个具体例子，知识点 = "一次函数"，但同一个"一次函数"知识点下，能出的题千差万别：

| 题型场景 | 能力要求 | 同样是"一次函数"知识点，考的东西完全不同 |
|---|---|---|
| 给解析式求图像经过象限 | 理解 | 考概念识别 |
| 给两点求解析式 | 掌握 | 考代数运算 |
| 一次函数与方程组联立 | 综合应用 | 考跨知识点整合 |
| 实际问题中的最优方案 | 建模 | 考数学建模素养 |
| 与几何动点结合的压轴题 | 综合应用 + 分类讨论 | 考高阶思维 |

如果你的系统只按"知识点"标题目，学生在"一次函数"上答错了，系统不知道他到底是**概念没懂**、**计算不行**、**还是综合应用能力差**。后续推荐就只能盲推同知识点的题，可能推一堆他早就会的概念题。

所以一道题的"标签"必须三维：
- **知识点**（数学对象是什么）→ 一次函数
- **能力要求**（要做到什么认知层级）→ 综合应用
- **题型场景**（在什么命题情境下）→ 与几何动点结合

这三维的**笛卡尔积**才是"考点"。一道题挂到一个考点上，等于同时挂到了三个维度的标签上。

---

**三个维度分别从哪里来**

- **知识点**：从课标的"内容要求"里抽，再加上教研专家拆细。前面讲过的 DAG 节点。
- **能力要求**：直接复用课标的**核心素养 + 学业质量水平**。比如数学的六大核心素养 × 水平 1-5，组合出能力坐标。这部分有官方依据，不要自己造。
- **题型场景**：从近 5-10 年真题里归纳。这部分纯靠教研经验，AI 可以做聚类草稿（把相似题聚成一类场景），人工命名和审核。

**对你产品的实际意义**

1. **运营录入是三张表，不是一张表**：知识点表、能力要求表（直接抄课标，最省力）、题型场景表（最重的工作）。考点是这三张表的关联。

2. **AI 能加速的部分**：从真题集里反推题型场景的聚类、给题目自动打三维标签（候选标签，人工确认）、根据课标条目生成能力要求的候选映射。

3. **诊断粒度直接受益**：学生在"一次函数 × 综合应用 × 几何动点"这个具体考点上错，系统能精准说"你的几何动点综合应用弱"，而不是笼统说"你函数没学好"。这就是产品价值的具体体现。

**2. 课标和知识点的关联**

这里有一个**关键认知**：**课标里的"内容要求"和你的"知识点"不是 1:1 关系**。

为什么？因为课标的内容要求是**教学目标级别**的描述，颗粒度比知识点粗。举例：

课标内容要求：
> "理解一次函数的概念，能根据已知条件求一次函数解析式，能画出一次函数图像并探索其性质。"

这一条内容要求，在你的产品里要拆成至少 4 个知识点：
- 一次函数的概念
- 求一次函数解析式
- 画一次函数图像
- 一次函数的性质（增减性、与坐标轴交点等）

**关系**：1 条课标内容要求 → N 个知识点（典型 N = 2~6）

**数据模型建议**：

```
课标条目表
  - id, 学科, 版本, 主题, 单元, 编号, 原文, 关联核心素养

知识点表
  - id, 名称, 学科, 学段, 粒度层级
  - 关联课标条目 ID（一个知识点对应一条主要课标条目）
  - 关联核心素养（可多个）

知识点-知识点 前置关系表（DAG）
  - from_id, to_id, 强度（强/弱）
```

注意**知识点的前置关系（DAG）和课标条目的顺序不一样**——课标内容是按教学顺序列的，知识点 DAG 是按认知依赖。一个知识点的前置可能来自完全不同的课标条目（甚至跨章节、跨年级）。这就是为什么知识点 DAG 必须独立建模，不能简单从课标抽。

**3. 考点和知识点的关系，怎么组织考点**

回顾上一轮的结论：**考点 = 知识点 × 能力要求 × 题型场景**。

关系是**多对多**：

- 1 个考点可能涉及多个知识点（"二次函数综合压轴题"会同时涉及二次函数、一元二次方程、相似三角形等多个知识点）

**数据模型**：

```
考点表
  - id, 名称, 学科
  - 主关联知识点 ID（主题归属）
  - 主关联能力点 ID（核心素养 + 水平）
  - 题型场景描述

  - 考点 ID, 知识点 ID, 角色（主考 / 辅考）

题目表
  - id, 题干, 答案, 难度

```

**考点怎么从零组织**——这是 MVP 最大的教研工作量：

**5. 权重怎么在系统中体现**

权重不是一个数字，是**多维度的统计量**。需要分三类：

**A. 考点权重（在卷子里的重要性）**
- 数据来源：近 5 年真题的考点出现频次 + 平均分值占比
- 字段：考点 ID, 出现频次, 平均分值, 分值占比
- 用途：推荐系统选题时，**考纲权重高的考点采样概率高**

**B. 题型分布权重**
- 数据来源：真题题型结构（选择 / 多选 / 填空 / 解答 各多少题、多少分）
- 字段：题型, 题数, 分值, 占比
- 用途：模拟卷生成时按这个分布出题；日常练习中按这个分布平衡题型

**C. 难度分布权重**
- 数据来源：真题难度系数（一般教研给出 0.3 / 0.5 / 0.7 / 0.9 分档）
- 字段：考点 ID, 难度档位, 占比
- 用途：根据学生当前水平推合适难度的题

**数据模型**：

```
考点权重表
  - 考点 ID, 模板版本, 频次, 平均分值, 分值占比, 难度分布
```

**实际怎么算权重**：
- AI 把近 5 年真题（约 50-100 套）解析成"每题对应哪些考点"
- 程序统计每个考点的出现次数、分值
- 教研人员审核异常值（比如某考点统计为 0 次但凭经验应该有，说明 AI 标注漏了）

**展示给学生**：不要直接给学生看权重数字，而是**用视觉化方式表现**：
- 每个考点旁边一个小条形图："本考点近 5 年高考出现 4 次，平均占 8 分"
- 在做完一次练习的总结页：饼图显示本次练习的考点分布是否合理

---

**Mastery 是什么**

直译：**掌握度**。在我们这个产品里，特指：

> **某个学生，对某个知识点（或考点），当前的"会到什么程度"的量化表达。**

它是个**数值**，记录在 `MASTERY_STATE` 表的 `mastery_score` 字段里，取值范围 0-1。

---

**为什么需要这个数**

如果系统只记录"会 / 不会"（二元），会出现很多问题：

- 学生第一次做对一道题就标"会"？太草率——可能是蒙的
- 学生今天做对，三周不练后还算"会"？显然不该
- 学生在某知识点的简单题做对，难题做错，算会还是不会？

所以需要一个**连续的、动态变化的数值**来描述"掌握的程度"。这就是 mastery。

---

**Mastery 的语义对照**

我们前面定的内部连续值 → 外部 4 档枚举映射，大概是这样：

| mastery_score（内部） | mastery_level（对外展示） | 含义 |
|---|---|---|
| 0.0 - 0.2 | 未开始 | 还没接触或全错 |
| 0.2 - 0.5 | 学习中 | 偶尔做对，不稳定 |
| 0.5 - 0.85 | 熟悉 | 大部分做对，有薄弱环节 |
| 0.85 - 1.0 | 已掌握 | 稳定做对，可进入长间隔复习 |

这些阈值不是定死的，MVP 跑起来后会根据真实数据调整。

---

**Mastery 怎么变化**

它**不是答一题就跳一次**，而是按规则**累积演化**：

1. **做对加分**：基础题 +0.05，中等题 +0.10，难题 +0.15
2. **做错减分**：基础题 -0.15，中等题 -0.08，难题 -0.03（难题做错不该被重罚）
3. **时间衰减**：N 天不练自动按艾宾浩斯曲线衰减（如 7 天后 0.9 → 0.75）
4. **讲述加成**：费曼讲述达标 +0.15-0.2
5. **粗心错误**：不扣分（因为不是知识问题）

**注意**：这只是示例规则。MVP 阶段会用简化版（如固定加减），冷启动后期再用学习模型（如 BKT - Bayesian Knowledge Tracing 或 IRT - Item Response Theory）做精细化。

---

**Mastery 在产品里驱动什么**

这是它的**实用价值**，不只是个统计数字：

| Mastery 的值 | 驱动的行为 |
|---|---|
| 接近 0 | 推荐学习材料（不出题），或先做前置知识点 |
| 0.2-0.5 | 推中等难度的相关题，做巩固训练 |
| 0.5-0.85 | 推变种题、跨知识点综合题，**适合插入讲述题** |
| ≥ 0.85 | 进入艾宾浩斯长间隔复习，**不再主动推主线题** |
| 整体下游知识点未达 0.3 | 阻塞推荐（因为前置不稳） |

所以"mastery 加 0.15"这种操作，**不是给学生一个分数**，而是改变系统下一次该给他推什么。

---

**Mastery 不是什么（避免误解）**

- **不是分数**：跟学生成绩、卷面分没关系
- **不是排名**：不跨学生比较
- **不是诊断结论**：不告诉学生"你水平 0.7"——只给枚举档位"熟悉"
- **不是一次决定**：是持续演化的状态

学生看到的永远是"未开始 / 学习中 / 熟悉 / 已掌握"四个词，**不会看到 0.65 这种数字**。0-1 的数值是**系统内部用于推荐和诊断的"工作变量"**。

---

**与之前讨论概念的关系**

回到我们这套系统：

- `MASTERY_STATE` 是表，**每个学生 × 每个知识点一条记录**
- 一个 Goal Instance 下，学生覆盖了 200-400 个知识点 → 就有 200-400 条 MASTERY_STATE

| 方案 | 准确率 | 成本 | 体验 |

**Q5.3 AI 怎么推断根因——它有什么信号可用**

AI 在批量分析时能看到的信号有限：
- 题干 + 标准答案 + 学生答案
- 学生答题用时
- 该知识点的历史正确率
- 该考点的题型场景

**不能直接看到**：
- 学生的思考过程
- 学生是不是抄了
- 学生是不是当时分心了

所以 AI 的"根因推断"本质上是**基于"对错 + 时间 + 答案差异程度"的启发式判断**。要不要在题目中**增加可选的"分步作答"**（解答题分小问输入），让 AI 能看到中间步骤？

**Q5.5 变种题 vs 原题重做的策略**

错题如何"再出"是关键，三种思路：

- **原题重做**：最简单，但学生可能背答案，没用
- **同知识点变种题**：换数字、换题型场景，但保持知识点
- **根因导向题**：根据根因决定推什么——concept 错推前置题；method 错推变种题；comprehension 错推审题题

我建议**按根因分流**：

- methodological → 推变种题（题型场景相同，参数 / 表述变化）
- comprehension → 推同题型的审题训练题
- computational → 不进错题本，无后续
- time_pressure → 限时模式重做
- out_of_scope → 暂不重做

**默认做法（一次性提交一个最终答案）**：
- 学生在一个文本框里写完所有内容提交
- AI 看到的：要么对，要么错
- 如果错了，**AI 不知道学生是卡在第（1）问，还是第（1）做对但第（2）算错**

**分步作答（分两个层级）**：

**层级 A（轻量级，强烈建议 MVP 做）**：

**对你两阶段提交的肯定 + 落地要点**

设计本质：把传统"上传 → AI 直接评判"拆成 **上传 → OCR 解析 → 学生确认 → AI 评判**。

这避免了一类很严重的问题：AI 把"x²"识别成"x2"，然后判定学生答案错误，学生还莫名其妙——OCR 错误**直接污染了评判**。两阶段提交把这两件事解耦了。

**但有 2 件事必须在 MVP 就想清楚**：

**1. 这个流程会显著拖长 session 周期**

原来流程：
```
答题 25min → 提交 → 30-60秒等待 → 解析页
```

新流程：
```
答题 25min → 上传 → 30秒 OCR → 学生逐题确认（可能 3-5 分钟）→ 提交 → 30-60秒 AI 评判 → 解析页
```

学生确认环节加进来后，**一次完整 session = 25min 答题 + 5-8 分钟解析 + 5 分钟 OCR 确认 ≈ 35-40 分钟**。这超出了番茄钟的 25 分钟。

**建议**：把 OCR 确认 + AI 评判 + 解析阶段**都视为"番茄钟外"的时间**，明确告诉学生"番茄钟到点了，接下来是复盘时间，不计入专注计时"。这是体验设计上的小事但要早说清楚。

**2. 跟 LLM chat 改 OCR 内容需要约束，避免学生作弊**

学生跟 AI chat 修正 OCR 时，理论上可以说"你看错了，我其实写的是正确答案 2ⁿ-1"——把错的改成对的。

**建议**：
- chat 仅用于修正 OCR 识别错误（"这个符号是 ÷ 不是 +"）
- 不允许直接修改最终答案（提供"这是我写的"和"我重新拍一张"两个选项，禁止"我重新打字"覆盖）

- 一道题修正超过 N 次自动锁定（疑似刷分行为）

这不是为了刁难学生，是为了让"做错"是真实信号。否则 mastery 全是假数据，整个推荐系统失效。
